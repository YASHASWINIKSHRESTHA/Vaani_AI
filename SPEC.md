## 0. Track decision — why Code, not Audio or Doc

Section 5 says every track needs the same evals and honesty regardless of format. Doc-only submissions are the easiest to fake and the hardest to defend if Shivam asks a follow-up question ("show me the run"). Code is the only track where the artifact *is* the proof — the eval harness, the raw per-sample logs, and the .wav files are generated by something that had to actually execute. Given the brief explicitly says "we can tell the difference between a pipeline you actually built... and a nice write-up that was generated and never touched real audio," Track A is the low-risk, high-signal choice, and it plays to strengths: this is the same "build the real thing, ship README + tests + demo" pattern as prior take-homes (Nova, VeriLoan, Cars24).

The one-paragraph summary Section 7 asks for still gets written — it sits at the top of the README, not as a separate Track C doc.

## 1. Core design principle: model proposes, rules decide

TTS output is not "correct" or "incorrect" by construction — it has to be *checked*. So the pipeline is two layers, never one:

- **Propose layer**: the TTS model generates a candidate clip for a given (text, language, reference-voice) tuple. This layer is allowed to be probabilistic, occasionally bad, and retried.
- **Decide layer**: a deterministic gate, driven entirely by config thresholds (not eyeballing), that runs round-trip ASR, speaker-embedding cosine similarity, silence/clipping checks, and duration-sanity checks on every candidate. A clip only counts as a "pass" if it clears the gate. Failures are logged, not discarded or silently retried into a good-looking number.

This is the differentiator vs. a plain "I ran model X, here's a number" submission: the acceptance criteria are code, so nobody (including me) can quietly round a 6.2/10 up to "meets target."

## 2. Per-language model selection

| Language | Primary model | Why | Fallback | ASR for round-trip WER |
|---|---|---|---|---|
| English | **Chatterbox** (Resemble AI, MIT-licensed) | Fastest/most natural open-source English TTS available now; strong zero-shot cloning | CosyVoice2 (better streaming RTF) | Whisper large-v3 |
| Arabic (MSA) | **XTTS-v2** (Coqui) | Explicit MSA support in its 17-language set, zero-shot cloning carries over | Fish-Speech (if its Arabic checkpoint proves usable) | Whisper large-v3 (weaker on Arabic — this is disclosed, not hidden) |
| Hindi / Indic | **AI4Bharat Indic Parler-TTS** | Purpose-trained on Indic prosody, not a multilingual model doing Hindi as an afterthought | AI4Bharat Indic-TTS as baseline comparison, XTTS-v2 as a third reference point | **IndicWhisper**, not Whisper large-v3 — Whisper's Hindi WER is meaningfully worse and using it would understate real intelligibility |

Router is a single `LANGUAGE_MODEL_MAP` in config, not if/else scattered through code — swapping a model for a language is a one-line change.

Known asymmetry to state up front, not discover late: CosyVoice2/Chatterbox/IndexTTS-2 barely cover Arabic or Hindi. One model for all three will not win. This per-language router is the answer, and the submission says so explicitly rather than pretending a single-model solution was attempted and just happened to fail.

## 3. Speaker similarity

**SpeechBrain ECAPA-TDNN** speaker embedding model → cosine similarity between reference clip and generated clip. Open source, well-validated, works reasonably cross-lingual since it's modeling voice timbre, not phonemes. Report both the cosine number and a blind human A/B ("same speaker or different?") — the brief asks for both, and a single automated number without the human check is exactly the kind of unverifiable claim a reviewer will discount.

## 4. Metrics, targets, and — critically — expected reality

| Metric | Target (per brief) | Realistic expectation |
|---|---|---|
| MOS | ≥4.0 | English likely clears it; Arabic/Hindi may land 3.3–3.8 |
| Speaker cosine | ≥0.75 | English/Arabic (XTTS/Chatterbox both good cloners) likely clear; Indic model may trail |
| Latency to first audio | <500ms streaming / <2s batch | Depends entirely on disclosed hardware — report literally, don't round favorably |
| RTF | ≤0.5 | Batch models on a single T4 may sit closer to 0.6–0.9; say so |
| Round-trip WER | ≤10% | English easy; Hindi/Arabic harder, especially with numbers/names |
| Cross-language floor | none below above bars | **This is very likely to fail somewhere.** Flag it as the headline finding, not bury it. |

The brief itself hints this asymmetry is expected ("Holds up across languages" is framed as an open question, and Section 6 explicitly rewards "this didn't work and here's why" over inflated numbers). A submission where every language hits every target uniformly is a red flag to an ML hiring team who already knows open Indic/Arabic TTS trails English in 2026 — it reads as fabricated rather than measured. Report the gaps as the actual finding.

## 5. Anti-hallucination guardrails (build these into the code, don't just "try to be honest")

Discipline alone doesn't survive a 3am deadline crunch. These are structural, not aspirational:

1. **Every eval run writes a raw JSON log**: timestamp, git commit hash of the code that ran, exact HuggingFace repo + revision/commit hash of every model used, GPU name (captured via `nvidia-smi`, not typed by hand), and per-sample (not just averaged) scores. Any number in the final report must trace back to a line in this log.
2. **Per-utterance CSV, not just an aggregate table.** Minimum 10–20 test sentences per language, deliberately including: short sentences, long sentences, sentences with numbers, sentences with names, and (bonus) code-switched sentences. A results table computed only from an aggregate mean is exactly what's easy to fake; per-sample data is what's hard to fake.
3. **Every generated .wav is saved and included**, named systematically (`{lang}_{model}_{sample_id}.wav`), alongside the reference clip it was cloned from. No audio artifacts = automatic rejection per the brief's own stated bar.
4. **A mandatory `FAILURES.md`** — the submission literally cannot be marked complete without at least one documented, specific failure per language (a sentence that broke, a name mispronounced, a number that came out wrong). This makes honesty a checklist item, not a virtue you have to remember under time pressure.
5. **Thresholds live in a YAML config, and pass/fail is computed by a script**, not asserted in prose. Removes the temptation (conscious or not) to eyeball a borderline number as "close enough."
6. **The results table in the README is generated from the per-sample CSVs by a script**, never hand-typed. Prevents transcription drift where a real 3.9 quietly becomes a reported 4.0.
7. **Hardware and library versions pinned in a `requirements.txt` / `environment.yml` and stated in the README header** — unverifiable latency/RTF numbers with no disclosed hardware are worthless and read as copy-pasted.

Why hallucinated numbers get caught, concretely, so the "why" is explicit rather than just asserted:
- Suspiciously round or uniform numbers across 3 languages that are known to have very different open-source model maturity.
- No raw per-sample data behind an aggregate — can't be interrogated.
- No audio files, or audio files that don't match the claimed sample count.
- Latency numbers with no named hardware.
- Zero documented failures in a task that explicitly asks for them — absence of any failure mode across three languages and 6 metrics in a 6–10 hour build is itself implausible.

## 6. Repo structure

```
vaani/
├── README.md                 # top-of-file: 1-paragraph summary (Section 7 requirement)
├── SPEC.md                   # this file
├── config/
│   ├── language_models.yaml  # router: language -> primary/fallback model
│   └── thresholds.yaml       # section-3 targets, single source of truth
├── src/
│   ├── router.py             # propose layer: dispatches to the right TTS model
│   ├── gate.py                # decide layer: runs ASR/similarity/sanity checks, pass/fail
│   ├── tts/                  # thin wrappers per model (chatterbox.py, xtts.py, indic_parler.py, ...)
│   ├── asr/                  # whisper.py, indic_whisper.py
│   └── similarity.py          # ECAPA-TDNN embedding + cosine
├── eval/
│   ├── test_sentences/       # {en,ar,hi}.txt — the fixed per-language test set
│   ├── run_eval.py           # runs every sentence through every language's pipeline, writes raw logs
│   └── build_report.py       # regenerates the results table from raw logs — never hand-edited
├── logs/                     # raw JSON logs, one per run, git-committed
├── samples/
│   ├── reference/            # the voice(s) cloned from, openly licensed or your own voice
│   └── generated/             # every output .wav, systematically named
├── FAILURES.md               # mandatory, per-language documented failure modes
└── requirements.txt / environment.yml
```
